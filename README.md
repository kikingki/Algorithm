        from collections import Counter

    음식의 스코빌 지수를 담은 배열 scoville과 원하는 스코빌 지수 K가 주어질 때, 규칙대로 음식을 섞어서 모든 음식의 스코빌 지수를 K 이상으로 만들기 위해 섞어야 하는 최소 횟수를 계산하는 문제. 모든 음식의 스코빌 지수를 K 이상으로 만들 수 없는 경우에는 -1을 반환해야 하고, 길이와 원소의 크기가 매우 큰 수라는 걸 주의해야 한다. 우선순위 큐를 이용하면 자동으로 최소 힙 상태를 만들어준다. 그래서 첫번째 요소와 두번째 요소를 섞다가 첫번째 요소가 K를 넘기면 조건을 충족해서 반복을 종료한다. scoville을 다 섞은 후에 첫번째 원소가 K보다 작으면 -1을 반환했다. 풀이는 맞는 것 같은데 계속 시간초과로 실패해서 원인을 알아본 결과, PriorityQueue와 heapq는 동일하다. 하지만 PriorityQueue는 Thread-Safe를 확인하는 절차가 추가로 포함되어 있어 시간에서 10~30배의 차이가 벌어진다. 파이썬의 특성상 대부분 멀티 프로세싱을 사용하기 때문에 Thread-Safe를 확인하는 것이 큰 의미가 없다고 한다. 파이썬을 사용할 땐 최소 힙은 heapq를 사용하면 될 것 같다. 나중에 실무에서 멀티 스레딩까지 고려하려면 머리가 깨질지도 모르겠다.
  + #### [프로그래머스 실패율](../master/실패율.py)
    전체 스테이지의 개수 N, 게임을 이용하는 사용자가 현재 멈춰있는 스테이지의 번호가 담긴 배열 stages가 매개변수로 주어질 때, 실패율이 높은 스테이지부터 내림차순으로 스테이지의 번호가 담겨있는 배열을 반환하는 문제. 이 문제도 딕셔너리 자료형을 이용해서 실패율을 담는 fail_late와 각 스테이지별 클리어 하지 못한 유저 수를 담는 fail_user를 만든다. 근데 각 스테이지별로 클리어한 플레이어 수를 구하는 방법이 떠오르지 않아 for문을 추가해서 하나씩 다 계산을 했는데 너무 비효율적인 방법이었다. 다음 스테이지에 도달한 플레이어 수는 실패한 플레이어 수를 빼면 되는 간단한 문제였는데 캐치하지 못해 아쉽다. 그리고 특정 기준으로 정렬을 하려면 람다식을 이용하는 걸 먼저 생각해보는 습관을 들여야겠다.
  + #### [프로그래머스 타겟 넘버](../master/타겟넘버.py)
    사용할 수 있는 숫자가 담긴 배열 numbers, 타겟 넘버 target이 매개변수로 주어질 때 숫자를 적절히 더하고 빼서 타겟 넘버를 만드는 방법의 수를 구하는 문제. 연산하는 숫자가 하나씩 늘어날 때마다 더하거나 빼는 경우의 수가 각각 2개씩 늘어나므로 트리 구조의 형태를 띈다. 트리 구조의 형태는 재귀로 구현할 수 있다. 더하거나 빼는 연산을 재귀로 호출하면서 연산 횟수를 1씩 늘려가다가 연산 횟수가 numbers의 개수일 때 연산의 결과 값이 타겟 넘버인지 확인한다.
